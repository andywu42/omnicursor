# OmniClaude Nodes and Contracts

How nodes and contracts fit into the automation pipeline — what they are, how they relate to tickets, and where they gate merges.

---

## Nodes

A **node** is a thin runtime component — not an agent, not a skill. It does exactly one operation. Every node has four parts:

```
node_name/
├── contract.yaml    ← declarative spec: inputs, outputs, Kafka topics
├── node.py          ← ~20-50 line shell that delegates to handlers
├── models/          ← Pydantic input/output types
└── handlers/        ← the actual business logic
```

### Node Types

| Type | Purpose |
|------|---------|
| **Compute** | Pure data processing, no side effects, deterministic |
| **Effect** | External I/O — REST APIs, Kafka, databases |
| **Orchestrator** | Multi-step workflow coordination |
| **Reducer** | State machine transitions |

### Skills vs Nodes

Skills are instructions to Claude ("read this file, run this command, call this agent"). Nodes are ONEX runtime units invoked via:

```bash
onex run node_name --input '{"ticket_id": "OMN-1234"}' --timeout 300
```

In the pipeline, a skill like `/ticket-pipeline` invokes the actual `node_ticket_pipeline` node to do the FSM work. The skill orchestrates; the node executes.

### Generating Nodes

The `/generate-node` skill uses ContractInferencer + LLM to produce the full four-file structure in ~10–25 seconds, including contract, business logic, tests, and docs.

## Contracts

There are two things named "contract" in OmniClaude. They are distinct.

### 1. ModelTicketContract

A structured YAML block auto-generated by `create_ticket` and embedded in every Linear ticket description. Also stored separately at `$ONEX_CC_REPO_PATH/contracts/{ticket_id}.yaml`.

It declares what proof is required before the ticket can be closed:

```yaml
schema_version: "1.0.0"
ticket_id: OMN-1234
summary: "Add retry logic to HTTP client"
is_seam_ticket: false          # touches Kafka/API/cross-repo boundary?
interface_change: false
interfaces_touched: []         # topics, events, protocols, public_api
contract_completeness: stub    # stub | enriched | full
dod_evidence:
  - check_type: command_output
    command: pytest tests/test_http_client.py -v
    assertions: ["PASSED"]
evidence_requirements: [unit, ci]
```

**Completeness levels:**

| Level | Meaning |
|-------|---------|
| `stub` | Auto-generated, minimal context, baseline DoD only |
| `enriched` | Stub + ticket-specific DoD extracted from description |
| `full` | Seam ticket with complete interface and cross-repo analysis |

**Seam detection** is keyword-based. If the ticket title or description contains "kafka", "topic", "api", "schema", "endpoint", etc., the ticket is flagged `is_seam_ticket: true`. This bumps evidence requirements from `[unit, ci]` to `[unit, ci, integration]` and demands cross-repo validation before merge.

Contracts are idempotent — if one already exists on a ticket it is not overwritten; data may only be added, never removed.

### 2. Node contract.yaml

Lives inside each node directory. Declares the node's runtime interface:

- Input and output schema
- Kafka topics produced and consumed
- Handler registrations
- Dependencies

This is what `contract_sweep` monitors for drift and runtime compliance.

## How They Fit Into the Pipeline

```
create_ticket
    └─ generates ModelTicketContract → embedded in Linear ticket description
           │
           ▼
ticket-pipeline PRE_FLIGHT
    └─ loads contract from Linear ticket
           │
           ▼
ticket-work IMPLEMENT
    └─ reads dod_evidence[] to know what tests to write
           │
           ▼
ticket-pipeline PR_REVIEW
    └─ if is_seam_ticket → runs contract-compliance-check pre-gate
           │
           ▼
ticket-pipeline AUTO_MERGE → CDQA gate
    ├─ dod_verify: runs each check in dod_evidence[]
    │              (file exists? tests pass? API assertions match?)
    └─ contract_sweep --runtime: verifies deployed handlers and topics
                                  match what contract.yaml declares
           │
           ▼
       merge allowed
```

---

## The CDQA Gate

CDQA is the set of quality gates that must pass before any merge is allowed. It combines two checks:

### dod_verify

Runs each item in the ticket's `dod_evidence[]` array:

| check_type | What it verifies |
|------------|-----------------|
| `file_exists` | A specific file was created |
| `pr_merged` | A linked PR was merged |
| `ci_green` | CI passed on the branch |
| `command_output` | A command produces expected output (e.g. `pytest` passes) |
| `rendered_output` | A rendered artifact matches expected content |

Produces a verification receipt. All checks must pass for the ticket to close.

### contract_sweep --runtime

Verifies the live deployed system matches what the node's `contract.yaml` declares:

- Registered handlers are present
- Subscribed Kafka topics exist and match declared patterns
- Published events match declared schemas

Exit codes: `0` = PASS, `1` = FAIL (creates a Linear ticket), `2` = QUARANTINE (infra down — warn only, do not block release).

---

## contract_sweep: Nightly Health Check

`contract_sweep` runs in `/onex:session` Phase B as part of the quality gate, in two modes:

### Drift Mode (default)

Scans all `contract.yaml` files across repos, computes SHA256 hashes, compares against pinned baseline snapshots.

Change classifications:

| Type | Examples | Action |
|------|---------|--------|
| BREAKING | input_schema change, algorithm change | Create Linear ticket, block |
| ADDITIVE | New optional fields | Create Linear ticket, warn |
| NON_BREAKING | Documentation updates | Log only |

Also validates Kafka boundary YAML staleness — checks that producer/consumer files still exist and topic regex still matches.

Deduplication key: `(repo, contract_path, drift_type)` — repeated drift on the same node does not create duplicate tickets.

### Runtime Mode (--runtime)

Verifies the live deployed system against all registered contracts. Runs the full 52-contract suite with `--all`. Auto-closes tickets when verification produces PASS for 2 consecutive runs.

Output: `.evidence/{ticket_id}/runtime-report.json`

## Full Relationship Map

```
Linear Ticket
  └─ ModelTicketContract (embedded YAML)
        │
        ├─ dod_evidence[]      ←─ dod_verify runs these at CDQA gate
        ├─ is_seam_ticket       ←─ triggers integration evidence requirement
        └─ interfaces_touched   ←─ integration_sweep validates cross-repo

Node Directory
  └─ contract.yaml
        │
        ├─ input/output schema  ←─ contract_sweep drift detects breaking changes
        ├─ kafka topics          ←─ contract_sweep runtime verifies live match
        └─ handler registration  ←─ contract_sweep runtime checks deployed system
```

---

## Key Invariants

- **Every ticket gets a contract** — even title-only tickets get a stub (auto-generated by `create_ticket`).
- **Seam tickets have harder gates** — `is_seam_ticket: true` requires integration evidence and cross-repo validation before merge.
- **Drift detection is automated** — `contract_sweep` runs nightly, creates tickets for breaking changes without human involvement.
- **Runtime verification is continuous** — `contract_sweep --runtime` runs in Phase B of every nightly session. Infra-down (exit 2) does not block release; actual contract mismatches (exit 1) do.
- **Contracts are append-only** — completeness can increase (stub → enriched → full) but fields are never removed.
- **Nodes are generated, not hand-written** — `/generate-node` produces the full four-file structure from a description; the contract.yaml is the source of truth for everything downstream.
